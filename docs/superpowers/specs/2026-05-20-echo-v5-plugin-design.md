# Echo v5 Protobuf Plugin Design

## Status

Accepted for planning on 2026-05-20.

## Context

This repository will provide Buf/protoc plugins that adapt `connect-go` generated services to HTTP frameworks. The first plugin is `protoc-gen-echo-v5`.

The plugin must not invent a duplicate REST annotation model. Existing projects already use:

- `google.api.http` from `google/api/annotations.proto` for REST method, path, body, and additional bindings.
- `buf.validate` for request validation rules.
- `gnostic.openapi.v3` for OpenAPI document and operation metadata.

`protoc-gen-echo-v5` should consume the Google HTTP annotations and generate Echo v5 REST bindings while preserving a Connect endpoint for clients that speak Connect/gRPC/gRPC-Web.

## Goals

- Generate Echo v5 registration code from proto services.
- Use `google.api.http` as the REST route contract.
- Reuse `protoc-gen-connect-go` generated service handlers and HTTP behavior.
- Keep request binding, response writing, validation, and error conversion replaceable at runtime.
- Put framework-neutral behavior in reusable packages so future Gin, Fiber, Kratos, or net/http plugins can share it.
- Support configurable generated file layout through normal Buf/protoc options.

## Non-Goals

- Do not define a custom HTTP path option for REST routing.
- Do not infer fake REST paths from Connect RPC paths.
- Do not hard-code project-specific form, response envelope, or error body policies into generated files.
- Do not implement full OpenAPI generation. Existing `gnostic.openapi.v3` annotations may be preserved for other tools.

## Project Layout

```text
bufplugins/
  buf.yaml
  buf.gen.yaml
  go.mod
  cmd/
    protoc-gen-echo-v5/
      main.go
  internal/
    generator/
      echo_v5.go
      google_http.go

  runtime/
    httpadapter/
      binding/
      errors/
      response/
      transcode/
      validate/
    echoadapter/

```

No custom option proto is required for REST routing. The repository can still include examples that import `google/api/annotations.proto`, `buf/validate/validate.proto`, and `gnostic/openapi/v3/annotations.proto`.

## Buf Configuration

The repository should be a normal Buf module:

```yaml
version: v2
modules:
  - path: proto
deps:
  - buf.build/googleapis/googleapis
  - buf.build/bufbuild/protovalidate
```

Examples that use OpenAPI annotations should document that `gnostic/openapi/v3/annotations.proto` is an input concern for OpenAPI tooling. `protoc-gen-echo-v5` does not need to link or interpret the Gnostic extensions.

User projects generate code with Buf `out` as the output root and plugin options for relative layout:

```yaml
version: v2
plugins:
  - local: protoc-gen-go
    out: gen/go
    opt:
      - paths=import
      - module=example.com/project/gen/go
  - local: protoc-gen-connect-go
    out: gen/go
    opt:
      - paths=import
      - module=example.com/project/gen/go
  - local: protoc-gen-echo-v5
    out: gen/go
    opt:
      - paths=import
      - module=example.com/project/gen/go
      - runtime_import=github.com/czyt/bufplugins/runtime/echoadapter
```

Supported plugin options:

- `paths=import|source_relative`, matching `protoc-gen-go` layout behavior.
- `module=<go import prefix>`, trimming the module prefix when `paths=import`.
- `file_suffix=.echo.pb.go`, default generated file suffix.
- `runtime_import=<go import path>`, Echo runtime import path.
- `connect_package_suffix=connect`, suffix used by `protoc-gen-connect-go`.

The plugin must not write outside the Buf/protoc output root.

## Proto Contract

REST routes come from `google.api.http`:

```proto
rpc SayHello(SayHelloRequest) returns (SayHelloResponse) {
  option (google.api.http) = {
    get: "/helloworld/{name}"
  };
}
```

The generator must support:

- `get`, `put`, `post`, `delete`, `patch`.
- `custom` HTTP methods.
- `body`, including `*` and named request fields.
- `response_body`, including `*` and named response fields.
- `additional_bindings`.
- Path variables such as `/users/{id}` and nested selectors such as `/users/{user.id}`.

Methods without `google.api.http` still get the Connect endpoint but do not get an Echo REST route.

## Generated Code Shape

For each proto file with services, the plugin generates one file in the same Go package as `protoc-gen-go` output:

```go
// Code generated by protoc-gen-echo-v5. DO NOT EDIT.
package userv1

func RegisterUserServiceToEcho(
	e *echo.Echo,
	svc userv1connect.UserServiceHandler,
	opts ...echoadapter.Option,
) error
```

The generated file contains static service and method specs:

- Go service and method names.
- Connect service path and method path.
- REST route method and Echo path.
- Request and response constructors.
- `google.api.http` body and response selectors.
- Whether the RPC is unary or streaming.

Only unary RPCs receive REST routes. Streaming RPCs remain available through the Connect endpoint.

## Runtime Boundaries

`runtime/httpadapter` is framework-neutral and should contain the reusable behavior:

- `binding`: path, query, JSON body, form, and multipart field binding into proto messages.
- `response`: response body selection, headers, status defaults, and proto JSON rendering.
- `errors`: bind, validate, transcode, and Connect error mapping into framework-neutral HTTP errors.
- `validate`: validator interface and default no-op implementation.
- `transcode`: request forwarding between REST-shaped HTTP requests and Connect handlers.

`runtime/echoadapter` is the Echo-specific adapter:

- Converts Google path templates to Echo path syntax.
- Registers Echo routes and middlewares.
- Adapts `echo.Context` to framework-neutral binding and response interfaces.
- Writes Echo responses and errors.
- Preserves Echo injection points such as logger, binder, validator, middleware, and route group.

## Request Binding

Generated code must not hard-code binding policy. It passes `MethodSpec` to a runtime binder:

```go
type RequestBinder interface {
	Bind(ctx context.Context, req RequestContext, spec MethodSpec, dst proto.Message) error
}
```

The default binder should support common REST behavior:

- Path variables bind to selected request fields.
- Query parameters bind to scalar, enum, message, and repeated fields that are not consumed by path or body.
- JSON body binds according to `google.api.http.body`.
- Form URL encoded and multipart fields bind through the same field selector rules.
- File upload handling is intentionally left to custom binders.

Default precedence is `path > query/form > body`. Conflicts should return a binding error unless the user installs a custom binder.

Runtime options must support global and per-method binders:

```go
echoadapter.WithRequestBinder(binder)
echoadapter.WithMethodRequestBinder("GreeterService", "SayHello", binder)
```

## Response Writing

Generated code passes response data to a runtime writer:

```go
type ResponseWriter interface {
	Write(ctx context.Context, resp ResponseContext, spec MethodSpec, src proto.Message) error
}
```

The default writer should:

- Use `google.api.http.response_body` when present.
- Return the full response message otherwise.
- Encode with proto JSON by default.
- Use status `200` for normal responses and `204` for empty selected response bodies.
- Allow custom status, envelope, header mapping, redirect, and content type behavior through runtime options.

Per-method response writers must be supported.

## Error Handling

Error handling is a separate runtime package, not a branch of response writing:

```go
type ErrorMapper interface {
	MapError(ctx context.Context, spec MethodSpec, err error) *HTTPError
}

type ErrorWriter interface {
	WriteError(ctx context.Context, resp ResponseContext, spec MethodSpec, err *HTTPError) error
}
```

The default mapper should distinguish:

- Request binding errors.
- Validation errors.
- Connect errors and Connect codes.
- Deadline and cancellation errors.
- Internal errors.

The default writer should produce a stable JSON error body with `code`, `message`, and optional `details`. Projects can replace the mapper or writer globally or per method.

## Validation

Validation is a runtime hook:

```go
type Validator interface {
	Validate(ctx context.Context, spec MethodSpec, msg proto.Message) error
}
```

The default validator is no-op. Projects using `buf.validate` can install the runtime adapter backed by `github.com/bufbuild/protovalidate-go` / `buf.build/go/protovalidate`. Connect interceptors may still perform validation on the Connect endpoint; Echo REST routes should have their own validation hook so binding errors and validation errors can become REST-shaped errors.

## Connect Reuse

Generated Echo registration should always register the Connect handler:

```go
basePath, connectHandler := userv1connect.NewUserServiceHandler(svc, connectOptions...)
```

The Connect endpoint is mounted into Echo as an HTTP handler for protocol-native clients.

REST routes for unary RPCs should call the generated service interface using the same handler implementation object. This keeps REST behavior simple and type-safe while preserving the Connect endpoint for protocol-native clients. Projects that need shared cross-cutting behavior should inject it through the Echo runtime hooks and Connect handler options.

## Generated Output Layout

Given:

```proto
option go_package = "example.com/project/gen/go/helloworld/v1;v1";
```

With:

```yaml
out: gen/go
opt:
  - paths=import
  - module=example.com/project/gen/go
```

The Echo file is:

```text
gen/go/helloworld/v1/helloworld.echo.pb.go
```

With `paths=source_relative`, the file follows the proto source path under `out`.

## Testing Strategy

Generator tests should use golden files for:

- Simple unary GET with path binding.
- POST with `body: "*"`.
- POST with named body field.
- `response_body`.
- `additional_bindings`.
- Method without `google.api.http`.
- Streaming methods.
- `paths=import`, `paths=source_relative`, and `module`.

Runtime tests should cover:

- Path/query/form/body binding precedence.
- Nested field selectors.
- Validation error conversion.
- Connect error code mapping.
- Response body selection.
- Echo route registration smoke tests.

## DI-Friendly Registration

Generated code should expose a small registrar object in addition to the convenience function:

```go
type UserServiceEchoRegistrar struct {
	Service userv1connect.UserServiceHandler
	Options []echoadapter.Option
}

func NewUserServiceEchoRegistrar(
	svc userv1connect.UserServiceHandler,
	opts ...echoadapter.Option,
) UserServiceEchoRegistrar

func (r UserServiceEchoRegistrar) Register(e *echo.Echo) error
```

This shape works with manual setup and common Go DI tools because the service implementation, options, and registration side effect are separated.
